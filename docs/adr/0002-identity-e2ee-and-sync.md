# ADR-0002：账号、2FA、端到端加密与多设备同步

- 状态：Accepted
- 日期：2026-07-14

## 目标

1. Web 后台首次启动时创建管理员账号，并绑定可由 Microsoft Authenticator 扫描的 TOTP 2FA。
2. 后续后台登录必须同时通过账号密码和 TOTP；管理员可创建、停用和重置 App 登录账号。
3. App 使用账号密码登录，不同账号的数据在本地、同步服务和推送通道中强隔离。
4. 对话、日程、提醒说明、SSH 备注等用户内容在进入服务器前完成端到端加密；后台管理员只能看到账号、设备、配额、同步状态和审计信息，不能读取内容。
5. 新设备完成身份登录和密钥解锁后，可同步已有对话与日程，而不是创建空账号。

## 管理员初始化与登录

- 管理后台不使用 GPT/ChatGPT 登录。仅当管理员表为空时，应用才允许创建唯一的 pending 管理员；数据库唯一约束、同源检查、来源限流和 10 分钟登记 Cookie 共同约束初始化竞争。部署者必须先通过受控网络完成首次初始化，再开放公网入口。
- 初始化分两步提交：
  1. 创建管理员用户名和强密码，密码使用随机盐、600,000 次 PBKDF2-SHA256 和服务端 pepper；数据库不保存明文或可恢复密码。
  2. 生成 TOTP secret 和 `otpauth://` URI，显示兼容 Microsoft Authenticator 的二维码；验证一个 6 位动态码后账号才激活。
- TOTP secret 使用服务端主密钥 AES-GCM 封装，二维码在浏览器本地生成，不调用第三方二维码服务。当前未实现恢复码，因此在恢复流程完成前，遗失 2FA 必须走受控的运维恢复，不能通过 App 账号绕过。
- 后续登录采用密码 + TOTP 两阶段挑战，使用短期、HttpOnly、Secure、SameSite=Strict 会话 Cookie；失败次数、锁定、退出、恢复和敏感操作均进入审计。
- 管理员会话有效期为 12 小时，并绑定 User-Agent 哈希；写操作要求精确同源请求。TOTP counter 以原子更新拒绝重放。

## App 账号与租户隔离

- 管理员可以创建账号、签发一次性临时密码、强制下次登录改密、停用账号、撤销设备和触发密码重置。
- App 密码使用与管理员不同的 pepper 域；登录返回 15 分钟访问令牌与 30 天刷新令牌，刷新时同时轮换两枚令牌。服务端仅保存令牌哈希；停用账号、后台重置密码或用户改密都会撤销全部旧会话。
- 所有服务端记录必须带不可复用的 `account_id`；每个查询、SSE 订阅、增量游标和资产 key 都在服务端做账号所有权校验，禁止依赖客户端传入的账号范围。
- 管理后台不存在读取用户内容的接口。后台只显示不可逆统计，例如密文条数、最近同步时间、密文字节数和设备状态。
- 日志、审计和错误信息不得写入密文解密结果、密码、TOTP secret、恢复码、访问令牌或完整远端命令输出。

## 内容加密与密钥模型

- 每个 App 账号在受信任客户端生成随机 256 位内容主密钥 `CMK`。消息、日程、提醒文本和其他敏感对象使用独立随机 nonce 的 AEAD 加密；AAD 固定包含 `account_id`、对象类型、对象 ID、版本和设备 ID，防止跨账号或跨对象替换。
- 服务器只保存版本头、最小同步元数据、nonce 和 ciphertext；不持有 `CMK`，也不提供解密端点。
- `CMK` 不直接由管理员管理的登录密码派生。否则管理员知道或重置登录密码时就能解密内容，与“管理员看不到内容”相冲突。
- 首个设备生成 `CMK` 后提供两种密钥恢复方式：
  1. 已授权设备通过一次性二维码/短码完成设备间密钥封装；
  2. 用户保存高熵恢复密钥，用它派生 KEK 后封装 `CMK`，服务器仅保存密钥信封。
- 新设备先完成账号密码登录，再通过已授权设备批准或恢复密钥解锁 `CMK`，随后下载并解密全部历史内容。登录密码重置不会自动授予旧内容解密权。
- 如果所有已授权设备和恢复密钥都丢失，管理员可以恢复账号登录权，但旧密文不可恢复；这是避免管理员托管解密密钥的必要安全后果。
- Rust `vault.db` 保存设备私钥、`CMK` 和密钥信封状态；Dart 继续拥有 `app.db`、领域模型和同步游标，遵守现有所有权边界。

## 实时同步

- 服务端维护仅追加的账号内加密变更日志，记录 `object_id`、`object_type`、`revision`、`operation_id`、`device_id`、密文和删除墓碑。
- 客户端使用单调游标拉取缺失变更，并通过账号绑定的 SSE 接收“有新 revision”通知；断线后始终以游标补拉为准，SSE 不作为唯一数据源。
- 写入采用幂等 `operation_id` 和乐观版本检查。日程等结构化对象使用字段级最后写入优先加显式冲突副本；对话消息采用不可变追加，避免覆盖。
- 新设备首次同步先获取密钥信封和快照，再按游标追平增量；解密后的日程由设备本地重新安排系统通知。
- 服务端可见的时间只保留同步和冲突处理所需值；标题、正文、对话、地点、SSH 备注及 AI 上下文始终位于密文中。

## 发布与验证门槛

- 独立部署开放公网前，后台路由必须完成应用层认证、CSRF 防护、速率限制、安全响应头和初始化抢占测试，并通过 TLS 提供服务。
- 必须覆盖：账号 A/B 越权、SSE 越权、重放、密码重置后旧密文、设备撤销、恢复密钥、新设备全量同步、离线冲突和通知重建。
- 密码学使用平台 Web Crypto 的 PBKDF2、HMAC、SHA-256 与 AES-GCM；内容同步阶段使用经过审计的 HKDF/AEAD 实现，不自行设计加密原语。未来切换 Argon2id 需要带算法版本的渐进迁移，不能使已有账号失效。
- 管理员能管理登录身份但不能恢复用户内容；若未来产品要求管理员无条件恢复内容，必须新增显式密钥托管模式，并在 UI 中清楚标示它会取消零知识保证。

## 当前实现边界（2026-07-15）

- 已实现应用自有管理员初始化、密码 + TOTP 登录、App 账号创建/启停/重置、App 账号登录、短期访问令牌、单次轮换刷新令牌、强制首次改密、限流、锁定和审计。
- 已实现的后台账号接口只返回账号状态、首次改密、锁定与最近登录元数据，不返回密码摘要或任何内容字段。
- 已实现服务端密文对象、追加事件、幂等写入、乐观版本、增量游标、账号隔离 SSE 与真实 MySQL 黑盒越权测试。
- Flutter 端 CMK、设备间 CMK 传递、恢复密钥、同步队列和新设备全量追平仍属于后续阶段；在这些能力完成越权与恢复测试前，不把“多设备实时同步”标记为端到端已交付。
